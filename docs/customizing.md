# Customizing for your taste

- `config/release-sources.yaml` — swap in publications, blogs, and label sites for discovery. Tier-1 sources are consulted every run; tier-2 only when their `genres` overlap that week's listening profile.
- `config/review-sources.yaml` — the review outlets whose endorsements decorate picks, plus the exact citation strings allowed in the email.
- `config/lastfm.yaml` — tune query periods, top-artist limits, and the similar-artist fan-out depth.
- `config/musicbrainz.yaml` — MusicBrainz verification + enrichment. After discovery, each kept candidate is resolved against the open [MusicBrainz](https://musicbrainz.org) database to confirm it exists and check its release-group first-release-date — confirming genuinely-new releases, demoting reissues, and flagging picks too new to have an entry yet (kept, never dropped). It's a curation *signal*, not a filter, and never a citation source. `enabled` is the master on/off switch (default on; set `false` to disable); `min_score` is the match-score floor. Two Phase 2 switches add one paced lookup each per resolved candidate (default on): `enrich_labels` reads the authoritative label/imprint (preferred over web-scraped prose when found), and `enrich_credits` reads the producer/engineer/collaborator credits, annotating picks whose personnel overlap artists you already play and measuring how well-populated new-release credits are. **One setup step:** add `musicbrainz.org` to your routine's Custom Network-access allowlist (no secret needed). Until you do, the step harmlessly no-ops and runs proceed exactly as before.
- `templates/email.html` and `templates/email.txt` — edit the email scaffold and copy. Keep the `{{placeholders}}` aligned across both files.
- **Model.** The model is set on the routine itself — Sonnet is the default (sufficient curation for this digest at a fraction of the token cost), with Opus available for deeper curation and Haiku for cheaper, faster runs. The `model:` frontmatter in `SKILL.md` is ignored by routines; it documents the intended default (see the [Claude Code skills docs](https://code.claude.com/docs/en/skills) for what the field means). Routines expose no effort control, so there's nothing to set there.
- `SKILL.md` prompt body — the orchestration itself is editable. Add a section, tighten the rubric, or change what gets logged.

## Providing feedback

The digest steers toward your taste over time from a single trusted file, `feedback.md` — append-only prose where you react to each week's picks (what you loved, want more of, or want pulled back, by artist, genre, or scene). Each Friday run reads it before searching (via `scripts/feedback.sh read`), weights the last ~12 weeks most heavily, biases its research toward what you've liked and away from what you haven't, and notes the influence in that run's `candidates.md`. An empty or missing file is fine — runs proceed normally until you start adding reactions. This is the *explicit* half of the feedback loop; the *implicit* half ([#25](https://github.com/mattroe/new-music-fridays/issues/25)) runs alongside it — see [How play-back steers the picks](#how-play-back-steers-the-picks) below.

**Where it lives.** `feedback.md` holds personal taste reactions, so it lives in your **private state repo** (`new-music-fridays-state`) — the same repo that holds run history and published digests — not in this public code repo. This repo carries only `config/feedback.example.md` (the format; copy it into the state repo as `feedback.md` to start). The state repo is the optional persistent store from [setup](setup.md#durable-run-history); if you haven't wired one up, runs still send normally, just without feedback steering.

Two ways to add a reaction, both landing in the same state-repo file:

- **Edit it directly.** In the state repo, append a bullet under a `## YYYY-MM-DD` heading and commit:
  ```markdown
  ## 2026-06-05
  - Bill Orcutt's latest album was great — more solo / exploratory guitar like that.
  ```
- **Tell the run.** Reopen the week's run (Routines → New Music Fridays → Runs → that run) and react in the conversation — the email footer reminds you of this. The session distills the steer, shows you the exact line it will add, and after you confirm, opens a PR against the **state repo's** `feedback.md` for you to merge (one click). It commits to a `claude/feedback-*` branch and the merge is a human gate. Note the tradeoff ([#35](https://github.com/mattroe/new-music-fridays/issues/35)): the state repo allows unrestricted pushes (so the routine can append history), so this gate is convention rather than branch protection — accepted as low residual risk. See `SKILL.md`'s "Capturing feedback (post-run)" for the protocol.

Keep `feedback.md` to taste signal about the picks only; questions and unrelated asks stay in conversation.

### How play-back steers the picks

Alongside your stated reactions, each run also reads your *behavior*: it takes the picks it sent you over recent weeks (from the [#17](https://github.com/mattroe/new-music-fridays/issues/17) history corpus in your state repo) and asks Last.fm whether you actually played them. One `get_album_info` call per release returns your album playcount and its track count, and the ratio between them buckets each past pick — *played-strong* (you replayed it), *played* (you listened through about once), *sampled* (you started but didn't finish — **not** counted as a thumbs-up), or *not-played*. Those buckets fold into the same curation steer as the explicit feedback: it leans toward the scenes you played and gently away from the ones you skipped or abandoned. It's a steer, not a filter — discovery still happens broadly — and your explicit feedback wins on conflict (stated taste outranks one noisy week of listening). Nothing extra to set up: it reuses the same state repo and the same history read, and with no history yet it's a silent no-op. Tune the lookback depth and the finished/repeat thresholds in `config/lastfm.yaml`'s `playback_lookback` block.
