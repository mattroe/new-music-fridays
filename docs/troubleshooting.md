# Troubleshooting

- **Pre-send validation aborts with a `from`/`to`/`subject` mismatch.** Check `config/delivery.yaml` — the values must match exactly what the prompt is about to send. Inline YAML comments on the same line as a value can trip naive comparisons, so keep comments on their own lines.
- **Routine sends nothing.** First check `config/delivery.yaml`: if `method: none`, that's by design — no email is sent; the digest is published to your state repo's `digests/<date>/` instead (see [Other delivery options](delivery.md)). Under `method: resend`, the send uses `scripts/send-email.mjs` — confirm `RESEND_API_KEY` is set on the routine's environment. The script prints the Resend error and exits non-zero on failure, so check the run's session transcript.
- **Cloud routine aborts validation with empty `from`/`to`.** The fresh clone is missing `config/delivery.yaml` (it's gitignored). Either commit it to your private repo (`git add -f`), or set `NMF_FROM`/`NMF_TO`/`NMF_SUBJECT` env vars so `scripts/write-delivery.sh` (run by `SKILL.md`) materializes it. Don't put this in the environment's *setup script* — that runs before the repo is cloned, so `config/` doesn't exist yet.
- **"Tool not found" errors during the run.** Confirm Last.fm is enabled on the routine's Connectors tab. The connector may register under a friendly name or a UUID prefix — `SKILL.md` matches by function-name suffix so either form works. (Resend isn't a tool — it's the `scripts/send-email.mjs` send.)
- **Resend rejects the send.** Verify your sending domain's DNS has propagated (Resend's dashboard will tell you) and the `from` address matches a verified domain or Resend's sandbox sender. Resend rejects "Name &lt;email&gt;" display-name wrappers in `from`; pass a plain address.
- **Send fails and the run log shows `send_error=host-not-allowlisted` (a `403 Host not in allowlist`, or a DNS-resolution failure / `ENOTFOUND` for `api.resend.com`).** This is an *environment* problem, not a Resend or code one: `api.resend.com` isn't on the routine's Network access allowlist, so the egress either refuses it (403) or the host doesn't resolve (DNS). **Re-running won't help** — fix the environment: edit the routine environment → Network access → Custom (the "Allowed domains" box), add a `api.resend.com` line, and check "Also include default list of common package managers" (add `musicbrainz.org` too if MusicBrainz is enabled). `scripts/send-email.mjs` tells these apart from a *real* Resend 4xx (which returns JSON) and from a transient blip (timeout/reset, left unflagged), and emits the `host-not-allowlisted` marker only for the not-reachable case. On the routine-test loop this surfaces as a `config-fail` comment on the PR, distinct from `transient-fail`. (Issue #66.)
- **Run aborts with a `spotify_error=` marker (Spotify taste backend, `config/taste.yaml::source: spotify`).** The abort is by design — a dead taste signal must not ship a generic, unpersonalized digest. The marker names the fix: `auth-failed` means the refresh token was revoked/expired or the client credentials are wrong — re-mint per [Spotify as the taste source](setup.md#optional-spotify-as-the-taste-source) and update the routine's `SPOTIFY_*` env vars; `host-not-allowlisted` means `accounts.spotify.com` and/or `api.spotify.com` are missing from the routine's Network access allowlist (both are needed — same fix pattern as the Resend entry above; re-running won't help); `profile-unavailable` means the core top-artist reads failed, likely a transient Spotify-side problem — a re-run is reasonable.
- **Run now sent a real production email.** That's expected — the production routine runs in production mode. For smoke tests, use a separate test routine with `NMF_TEST=1` set — see [Testing a fork](testing.md#testing-a-fork).
- **`meta.json` shows `sent: false` or `sent: null`.** `false` means a send was attempted and failed — either pre-send validation failed (look for the abort message in the run log) or the send call itself errored; the run-directory artifacts are still useful for debugging. `null` is *not* a failure: it means `method: none`, so no send was attempted — look in `digests/<date>/` in your state repo for the delivered digest.
- **`meta.json` shows `tokens: null`.** Expected — a routine run can't read its own token usage. Review per-run usage in the run's session transcript, and aggregate spend at [claude.ai/settings/usage](https://claude.ai/settings/usage).
- **Run history isn't being saved (`meta.json.notes` mentions `history not persisted`).** Persistence is best-effort and never blocks the send, so the email still arrives. Check the `reason`: `state-repo-not-found` means no state repo is set up or the routine isn't cloning it (see [Durable run history](setup.md#durable-run-history), or set `NMF_STATE_DIR`); `git-push-failed` usually means "Allow unrestricted branch pushes" isn't enabled on the state repo, so the push to `main` was rejected (enable it, or set `NMF_STATE_BRANCH=claude/history`). Only production runs persist — test runs are excluded by design.
